3｜架構圖：CMS 接在哪個位置

CMS 只取代「編輯入口」：資料嚴格內網、經 API 匯出進既有 pipeline，snapshot／draft／gate 30 規則位置不變。

鐵律｜編輯 ≠ 發布：機器閘（gate 30 規則）是唯一上線關卡——CMS 只到 export，發布通道隔離；資料嚴格內網。
邊界：不做拖拉自由版型。版型受 02 號模板契約管控，是一致性優點而非限制；CMS 只管「欄位內容」，不開放自由排版。

4｜建置時間軸：Phase 0～4

先解前提（N-13／PF-11），再用約 2 週 spike 取證，裁決後才進建置與治理；每階段都有明確驗收。

Phase 0

前提工程

  • N-13 loader：?draftUrl= 預覽端點（視覺化路線總前置）
  • PF-11 資安確認：Workspace 邊界／內網主機 IT owner
既有佇列：N-13 既有佇列：PF-11
驗收：預覽端點可用；內網主機與 owner 確定。
Phase 1

Spike（建議 2 週框）

  • Directus 內網試裝
  • 03 Schema→內容模型映射 PoC（鈦案單案）
  • 欄位級權限 🔒✍️🎚 真鎖實測
  • Live Preview 接 FrameTest「改→看」延遲實測
  • Visual Editing 可行性（field-map 埋點）
  • 與 B-i2 Grist PoC 並列評測
驗收：評測表（權限真鎖／gate 前置強制性／運維 owner／成本）。
Phase 2

裁決

  • Spike 報告提交
  • 裁決收斂會定編輯入口形態：本案 vs Grist vs 維持 Sheets
驗收：裁決紀錄成案（去向唯一）。
Phase 3

建置（若採納）

  • 內容模型全遷；03 Schema 雙向同步治理機制
  • pipeline 改讀 CMS API（取代 xlsx 匯入；snapshot／draft／gate 不變）
  • 變更單／審核流入 CMS workflow
  • L3 Visual Editing 埋點
驗收：鈦案 E2E 走測通過；CI schema 比對防漂移。
Phase 4

治理

  • 發布權限 R-5 裁決落地
  • 跨部門培訓與 SOP
  • 與 13 號 Stage C 薄工作台路線的取代／併軌決策
驗收：SOP 發布；R-5 與併軌決策成案。
