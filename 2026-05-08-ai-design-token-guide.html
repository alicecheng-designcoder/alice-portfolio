前言

AI 為什麼會「腦補」視覺風格？

你給了 AI 你的 design token，描述了品牌風格，甚至貼上了截圖——但 AI 生成的結果仍然充滿 Material Design 的影子，或者莫名出現漸層。這不是 AI 工具的問題，而是我們給 AI 的資訊類型根本不對。

AI 的腦補行為來自兩個根本原因：

A

Token 的語意距離

你提供了 design token 的數值，但 AI 無法「感知」這些數值背後的視覺意圖。當約束不夠精確時，模型會用訓練資料中的視覺慣例——Material Design、Fluent UI、通用 SaaS 風格——填補空白。

B

負向約束不足

告訴 AI「用這些顏色」，但沒告訴它「不能用這些顏色」。AI 會在已知範圍和訓練資料之間做混合，而不是嚴格限制在你的品牌範圍內。

現在給 AI 的 AI 需要的
截圖（視覺印象） 結構化數值（可執行規則）
設計文件截圖 元件對應清單（組裝指令）
口頭描述風格 禁止清單（負向約束）

「不要期待 AI 理解品牌，要讓 AI 沒有機會偏離品牌。」

01

七條 AI 協作工作原則

這七條原則不是使用技巧，而是根本性的工作思維轉換。 每一條都對應一個 AI 腦補的具體來源，從描述型改為約束型， 從頁面優先改為元件優先，從截圖輸入改為結構化數值。

P1

Prompt 從「描述型」改為「約束型」

✗ 不要寫：「幫我設計一個符合品牌風格的登入頁面」
✓ 要這樣寫：「背景色只能用 surface/basic/level1。按鈕只能用 Button/Primary 元件規格。不得新增任何未在 token 清單中的顏色。若不確定某元素的視覺處理，標註 [NEEDS DESIGN]，不要自行填補。」

最後一句最重要：教 AI 承認不確定，而不是腦補。

P2

元件優先，不是頁面優先

舊順序（容易腦補）：AI 想像整個頁面 → 套用 token
新順序（精準執行）：從元件庫組裝 → 只有佈局層需要新決策

Prompt 範例：「這個頁面由以下現有元件組成：Header（已有）、Card/Product（已有）、Button/Primary（已有）。你的工作只是決定這些元件的佈局排列與間距。間距只能使用 spacing token 清單中的數值。」

P3

每次 Prompt 前段貼入 Token 約束清單

不要只說「請參考設計系統」，而是把約束清單直接貼進 prompt 的最前段作為 system context。每次對話都需要重新貼入（AI 沒有跨對話記憶）。

Claude Code 的做法：在專案的 CLAUDE.md 加入設計系統規則，Claude Code 每次執行前會自動讀取，不需要每次手動貼入。

P4

建立品牌反模式清單（負向約束）

在設計系統文件中加入「不是什麼」比「是什麼」更有效。明確告訴 AI 禁止參考的視覺風格：不是 Material Design、不是 Fluent UI、不使用漸層作為主視覺、icon 只用 outline 不用 filled、不使用投影作為卡片區隔用 border 替代、圓角最大值為 radius/xl（24px），不超過此值。

P5

要求 AI 產出自我稽核表

在每次 prompt 結尾加上稽核要求：使用了哪些顏色 token（逐一列出）、有無任何不在清單內的視覺決策、標記為 [NEEDS DESIGN] 的項目。

這樣 review 時只需看稽核表，不用逐一核對整份設計。

P6

分階段產出，不要一次要求完整頁面

完整頁面的 prompt 給 AI 太多自由空間。改為四輪分階段產出：
第一輪 → 頁面資訊架構（IA），不含任何視覺樣式
第二輪 → 確認 IA 後，套用元件對應
第三輪 → 佈局與間距 token
第四輪 → 最終視覺 QA，對照 token 稽核

P7

Figma Variables 轉化為 AI 可讀格式

Figma variables 是給人看的，需要轉化成 AI 可精準執行的格式。安裝 Variables to JSON 或 Token Studio plugin，匯出 JSON 檔案，再基於 JSON 製作三份 AI 執行文件（見第 04 節）。

截圖給 AI 的問題：AI 看截圖是「用眼睛判斷顏色」而不是「讀取 token 名稱」，每次都在猜，無法精準繼承。

Claude Code 專案設定

在專案根目錄建立 CLAUDE.md，加入 Design System Rules 區塊，告訴 Claude Code 每次執行前先讀取 design-system-rules.md，遇到不確定的視覺決策時輸出 [NEEDS DESIGN: reason] 而不是自行猜測。這樣 Claude Code 每次執行前會自動讀取設計系統規則，不需要每次手動貼入。

02

Design Token 健檢結果

以下是針對五份 token 檔案的健檢結果——Component_tokens.json、Mode_1_tokens.json、dark_tokens.json、light_tokens.json、value_tokens.json，共 494 個 token。 這份健檢的目的不是評分，而是找出 AI 腦補最嚴重的缺口在哪裡。

做得好的地方

項目 說明 AI 可讀性
三層架構完整 Primitive → Semantic → Component，AI 可以沿層追蹤語意 優
Light / Dark 完全對稱 131 個 semantic token 兩模式完全對應，無缺漏 優
Radius 有情境描述 每個 radius token 都有 $description 說明適用元件 優
Component token 命名語意佳 button/surface/primary-default 結構清楚，AI 可直接對應 CSS 屬性 優

需要優化的地方

P0

嚴重 Typography token 幾乎缺失

Mode_1 只有 font-size 數值（10–48px），缺少 font-weight、line-height。Semantic 層沒有任何 type token（heading/lg、body/md 等）。影響：AI 在建置頁面時字型規格完全靠猜，這是腦補最嚴重的來源。

P0

嚴重 Spacing token 完全缺失

整個系統沒有任何 spacing / padding / gap token。影響：AI 建置元件時無法知道標準間距，畫面「密度感」與品牌不符。

P1

重要 Component token 覆蓋率只有 6 個元件

現有：button / tag / tab-chips / pagination / toast / searchbox。缺少：input field、card、modal、navigation、form 等常用元件。影響：AI 遇到這些元件只能回退到 semantic token，容易腦補視覺細節。

P2

次要 Semantic token 命名夾雜特殊符號

text--icon（雙 dash）、divider--border 在 CSS variable 轉換時需特殊處理。影響：AI 直接生成 CSS 時容易產生無效的 var(--text--icon) 變數名。

各類型 Token 的 AI 可讀性評估

顏色 Token

90%

三層架構完整，Light/Dark 對稱，AI 可精準引用

圓角 Token

92%

有 $description 說明適用情境，AI 可對應元件

03

Design Token 優化待辦清單

優化順序以「對 AI 腦補的影響程度」為依據，不是以「設計完整性」為依據。 P0 的兩項直接決定了 AI 能否產出可用的字型和間距規格；P1 決定高頻元件的精準度；P2 是長期系統品質。

P0：立即執行（影響最大）

Typography — 補充 Primitive
在 Mode_1_tokens.json 的 type 群組加入 font-weight regular → 400 ／ medium → 500 ／ bold → 700
立即
加入 line-height primitive tight → 1.2（標題）／ normal → 1.5（body）／ loose → 1.75（中文 body 建議 1.6–1.7）
立即
Typography — 建立 Semantic Token
新增 type_tokens.json，建立 Heading 語意 token xl / lg / md / sm，各含 size + weight + line-height，加上 $description 和 mobile-fontSize 備註
立即
建立 Body / Label / Caption 語意 token Body lg/md/md-strong/sm ／ Label md/sm（互動元素）／ Caption md/sm（說明文字）
立即
Spacing — 建立完整系統
在 value_tokens.json 加入 spacing primitive（與 radius 同層） 基礎單位 4px，scale：4 / 8 / 12 / 16 / 20 / 24 / 32 / 40 / 48
立即
建立 Spacing Semantic Token spacing/component/xs~lg（元件內距 4/8/12/16px）／ spacing/layout/xs~xl（版面間距 8/16/24/32/48px）
立即

04

AI 可讀文件製作：三份執行文件

設計系統完善後，需要製作三份「AI 執行文件」供 prompt 使用。 這三份文件的核心邏輯是：把設計系統的視覺語言轉化成 AI 可以精準執行的指令語言。

文件 A：Token 約束清單（最重要）

純文字格式，每次 prompt 前段貼入。包含強制使用清單（所有可用 token 名稱與數值）和禁止清單（不得自創的顏色、樣式、行為）。這份文件的存在，讓 AI 的工作從「視覺決策」縮減到「Token 選擇」。

文件 B：元件對應速查表

說明每種 UI 需求要用哪個元件，以及該元件的關鍵規格。

當頁面需要「主要操作按鈕」→ 使用 Button/Primary └ 高度: 44px, 圓角: radius/2xl, 色: surface/element/action/brand 當頁面需要「內容卡片」→ 使用 Card/Default └ 邊框: divider--border/basic/default, 圓角: radius/md, 無陰影 當頁面需要「文字輸入欄」→ 使用 Input/Default └ 高度: 48px, 圓角: radius/sm, border: input/border/default

文件 C：品牌反模式說明

列出 AI 禁止參考的視覺風格，搭配正面描述。「不是什麼」的清單有時比「是什麼」更有效——它直接堵住 AI 最常走的偏離路徑。

品牌反模式（AI 禁止參考）： ✗ 不是 Material Design 風格 ✗ 不是 Fluent UI 風格 ✗ 不使用漸層作為主視覺（品牌色是純色系） ✗ icon 不使用 filled 風格，只用 outline ✗ 不使用投影作為卡片區隔，用 border 替代 ✗ 圓角最大值為 radius/xl（24px），不超過此值

一次性建立步驟

安裝 Variables to JSON 或 Token Studio plugin → 匯出 JSON 檔案（設計系統更新時重新匯出）→ 基於 JSON 製作三份 AI 執行文件。之後只需在設計系統更新時同步匯出，不需要每次手動維護。

05

Prompt 撰寫範本與 Claude Code 設定

以下是可以直接複製使用的 Prompt 結構，以及 Claude Code 專案的設定方式。 這些範本的核心邏輯：把 AI 的工作定義得愈窄，結果的精準度愈高。

標準新頁面 Prompt 結構

── 系統規範（強制執行，不得偏離）──────────── （貼上 Token 約束清單 / ai_design_spec.md 完整內容） ── 元件清單 ────────────────────────────────── 本頁面使用以下現有元件（不需重新設計樣式）： - [元件名稱 1] - [元件名稱 2] ── 任務 ─────────────────────────────────────── 請根據以上規範，建置 [頁面名稱]，包含： - [功能區塊 1] - [功能區塊 2] 佈局：[描述排版方向與欄位數] 平台：[Mobile / Desktop / 兩者] ── 稽核要求 ─────────────────────────────────── 產出後請附上稽核表： 1. 使用了哪些顏色 token 2. 使用了哪些間距 token 3. 有無任何不在清單內的視覺決策 4. 標記為 [NEEDS DESIGN] 的項目

Claude Code 專案設定

在專案根目錄建立 CLAUDE.md，讓 Claude Code 每次執行前自動讀取設計系統規則：

## Design System Rules - Always read `design-system-rules.md` before generating any UI component or page - Never deviate from token values defined in that file - When uncertain about a visual decision, output [NEEDS DESIGN: reason] instead of guessing - Component assembly takes priority over visual creation

在同目錄建立 design-system-rules.md，放入 Token 約束清單（文件 A）、元件對應速查表（文件 B）、品牌反模式說明（文件 C）的完整內容。 這樣設定後，不需要每次對話手動貼入約束，Claude Code 會自動讀取。

「給 AI 的資訊不是愈多愈好，而是愈精確愈好。一份清晰的禁止清單，比十頁品牌故事更有效。」

維護節奏建議

本知識庫應隨 design token 更新同步維護。建議每季檢視 AI 協作原則是否需要根據工具更新調整——AI 工具每季都在演進，今天有效的約束方法，半年後可能需要調整。
