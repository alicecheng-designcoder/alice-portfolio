關鍵互動（「所見即所得」的具體定義）： 1. 模組↔欄位雙向定位：點預覽的任一模組→面板捲至對應欄位群並高亮（依 P2-4 field-to-UI matrix）；改欄位→預覽即時重渲染（FrameTest 資料層驅動、既有能力）。 2. 編修即預檢：欄位失焦即跑輕量檢查（字元上限表 03 §8.1、優惠詞掃描、必填）——紅字即時回饋；完整 gate 於「送核可」時執行。 3. 真值欄唯讀＋溯源：🔒 欄顯示值＋來源（actid/gdid 連結）；要改→按「重抓」由 Claude 代行更新，人不能手打數字（真值鐵律在 UI 層的落實）。 4. 理由行常駐：AI 每個選擇的白話理由（P3）顯示在欄位旁——跨部門同事不需要懂規則文件就能理解「為何這樣選」。

§3 資料流：Google Sheets＝單一真相源（Single Source of Truth）

發起：URL(+主題/目的) ──AI 策展──▶ 初稿 → gate → 寫入 Sheets（P1~P4）
                                                    │
預覽（讀路徑）：Sheets ──「發佈到網路」JSON/CSV（唯讀、零憑證）──▶ 平台預覽渲染
編修（寫路徑）：平台編修面板 ──寫回 API──▶ Sheets 對應儲存格（附 R5 註記）
                                   └─▶ 寫回前過欄位級預檢；送核可時過完整 gate
維護：營運也可直接在 Sheets 編輯（試算表原生協作）→ 平台預覽自動反映（讀路徑輪詢/重新整理）

雙入口原則：平台編修與 Sheets 直接編輯都允許（Sheets 是母體、平台是視覺化外掛）——平台不鎖死資料、Sheets 手改由指紋＋gate 於建置時攔截異常。這使跨部門同事可各取所需：習慣試算表的人留在 Sheets，需要視覺對照的人用平台。

§4 三階段路線（每階段皆可獨立交付使用）

Stage A：策展工作台 v0（近零後端，2–3 個建置日等級）

  • 形態：靜態站（沿用 Netlify）＋每活動一份資料 JSON。發起與 gate 仍由 Claude 代行（Blocking #2 MVP 裁決不變），產出＝xlsx＋預覽連結（titanium-preview 的通用化：/preview/{活動id}/）。
  • 編修：面板編修→變更暫存於瀏覽器→匯出「變更單」（xlsx 差異檔或逐欄清單）→ 由 Claude 代行/人工回寫 Sheets——尚非直連，但跨部門已可「看真視覺＋提編修」。
  • 讀路徑升級（A+）：Sheets「發佈到網路」→ 平台直讀（唯讀直連零憑證）——Sheets 改了、預覽就變，先把「連動」做到讀這一半。
  • 風險：寫回是人工斷點；公開 Netlify 的機密性（→§5）。

Stage B：半自動直連（GAS／service account，EP-3 規格＋輕開發）

  • 寫路徑接通：Google Apps Script Web App（綁 Sheets 本體、天然同域權限）承接平台的欄位寫回＋R5 註記＋欄位級預檢；完整 gate 由 Claude 代行排程或 GAS 觸發（gate 移植性評估＝EP-3 規格項）。
  • 對應 N-9「Sheets 直寫升級位」的落地形態之一；認證用 Google 帳號（跨部門即 Yahoo Google Workspace 帳號，權限＝Sheets 共用設定，零新帳號系統）。
  • R2 核可流轉在平台上操作（改狀態欄），發布仍 export-only（12 號文件裁決前提不變）。

Stage C：後台 v1（EP-4，Gate：M3 全過＋使用者確認）

  • 正式工作台：多活動列表、權限分級、審核流、版本 diff、發布 pipeline。本文件 §2 介面即其規格草圖；Stage A/B 的使用回饋作為 EP-4 需求輸入。

§5 治理與機密性（平台化的新增風險，需正視）

  1. 站點可見性：現行 Netlify 站公開無認證。內部價格策略/檔期若上平台，必須先裁決：(a) Netlify 密碼保護/Identity（低成本）(b) 內網部署 (c) 接受公開（僅 demo 資料）。建議 (a) 起步。
  2. 真值鐵律 UI 化：🔒 欄在任何階段都不開放手改（§2-3）；平台寫回一律走欄位白名單（✍️/🎚 清單由 Schema 權限欄生成）。
  3. 審計：所有平台寫回帶 R5 註記（誰/何時/原值→新值）；Sheets 版本歷史為兜底。
  4. 單一真相不分岔：平台永不持有獨立資料庫——JSON 一律由 Sheets/pipeline 產出，避免第二真相源。
